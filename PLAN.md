## 現状把握

### 全体構成
- エントリは `src/App.svelte`。
- タイムライン描画の中心は `src/lib/components/Timeline.svelte`。
- 再生状態・ノート配列・playhead・speed などの単一状態源は `src/lib/timeline/store.svelte.ts`。
- Note型は `src/lib/nostr/types.ts`。
- ノート画像抽出ロジックは `src/lib/nostr/images.ts`。

### 現状の各要望との対応状況

#### 1. 投稿クリックで njump を開く
- 現状、ノート本体クリックは `Timeline.svelte` の `openMenu(note)` に接続されている。
- クリックするとその場で外部遷移するのではなく、オーバーレイメニューを開く仕様。
- 要望の本質は **`https://njump.me/<note1...>` を別タブで開くこと**。単なる njump トップページ遷移ではなく、**対象投稿の `note1` を後ろに連結した個別投稿ページ遷移** が必要。
- したがって「njump導線が未実装」ではなく、**現状クリック挙動が要望仕様にまだ一致していない** と整理するのが正しい。

#### 2. 50倍速対応
- `src/App.svelte` の `SPEED_OPTIONS` は `[1, 1.5, 2, 3, 5, 8, 10, 15, 20]`。
- `timeline.setSpeed()` 側は `n > 0` の数値であれば受け入れるため、**状態管理層は20×固定ではない**。
- 現状の上限は実質的に **UIの選択肢が20×まで** で決まっている。
- 高速化に伴う副作用確認（playhead進行、TTS、視認性）は未整理。

#### 3. 投稿画像表示
- `Timeline.svelte` ではすでに `getNoteImageUrl()` を使って画像を描画している。
- モーダルにも画像プレビューがあり、画像クリックで別タブ表示できる。
- `src/lib/nostr/images.ts` では以下を検出している。
  - NIP-92 `imeta`
  - NIP-94系 `url` + `m` / `mime`
  - 本文中の直リンク画像URL
- つまり**「画像を表示できるように」は完全未着手ではなく、すでに一次実装済み**。
- ただし、現要望の意図次第では以下の不足がありうる。
  - 1枚のみ表示で、複数画像に未対応
  - 画像検出対象が限定的
  - レイアウト上、画像付きノートの横幅見積もりに画像寸法が反映されていない
  - 壊れた画像/重い画像/NSFW等の扱い未整理

#### 4. 横位置が時系列になっていない
- タイムライン上の横位置は `Timeline.svelte` で `f = (playhead - ms) / win` を計算し、`style="right: {p.f * 100}%"` で配置している。
- ノート配列 `timeline.notes` は `store.svelte.ts` で `created_at` 昇順に挿入され、`visibleNotes` もその順序を維持している前提。
- そのため、理屈上は **created_at が正しければ右（新）→左（旧）の時系列整列になる設計**。
- それでもズレが見える場合、原因は「並び順そのもの」よりも、**見た目上のアンカー位置・幅・同一時刻イベント・レーン予約ロジック** にある可能性が高い。

## 要望ごとの実装方針

### 1. 投稿をクリックすると `https://njump.me/` で別タブで開く

#### ギャップ
- 現状はクリックでメニューを開く。
- 要望はクリック時の主動作を「外部リンクを別タブで開く」に変えること。

#### 実装方針
- 方針A（本命）:
  - ノート本体クリックを `window.open(njumpUrl, '_blank', 'noopener,noreferrer')` に変更。
  - 既存メニュー機能は、別ボタン・長押し・右クリック相当・小さな `⋯` ボタンなどに退避する。
- 方針B（後方互換寄り）:
  - クリックは njump 遷移、キーボード操作や補助UIからメニューを開けるようにする。
- njump URL は **`https://njump.me/<note1...>` を基本仕様として固定**する。
  - つまり対象投稿の event id から **`note1` を生成して連結**する。
  - `nevent1` を使うかどうかは今回の要件では広げず、まずは **ユーザー指定どおり `note1` ベース**で計画する。
  - `nostr-tools` など既存ユーティリティで event id → `note1` 変換を行えるかを確認し、手書きエンコードは避ける。

#### 影響ファイル候補
- `src/lib/components/Timeline.svelte`
- 必要なら `src/lib/nostr/` 配下に njump URL 生成ヘルパー追加
- 必要なら `src/App.svelte`（説明文やヘルプ更新）

#### 注意点
- 既存の「全文表示」「TTSミュート」がクリック導線から消えるので、代替導線を同時に設計する必要がある。
- キーボード操作（Enter/Space）やアクセシビリティ文言も合わせて見直す。

### 2. 50倍速まで対応

#### ギャップ
- 現状UIは20×まで。
- state/store側は数値上限を持っていないため、UIと挙動確認が主論点。

#### 実装方針
- `App.svelte` の `SPEED_OPTIONS` を 50×まで拡張する。
- 追加候補:
  - 25×
  - 30×
  - 40×
  - 50×
- 50×にしたときの主要確認点:
  - `#step()` の `playheadMs += elapsed * speed` が期待通り進むか
  - seek/nudge/LIVE復帰が破綻しないか
  - TTSが高速再生時に実質追従不能でも破綻しないか
  - 画面上でノートが瞬間的に流れすぎて操作不能にならないか
- 必要であれば実装時に以下も検討:
  - 20×超ではTTS自動停止、または挙動説明を加える
  - 高速域だけ刻みを荒くする

#### 影響ファイル候補
- `src/App.svelte`
- 必要なら `src/lib/timeline/store.svelte.ts`（速度上限・TTSポリシー追加時）
- 必要なら `README.md`

#### 注意点
- 技術的には軽い変更だが、体験面では 20×→50× でノート通過速度がかなり上がる。
- TTSは「再生速度」と「発話速度」が連動していないため、期待差が出やすい。

### 3. 投稿に含まれる画像を表示できるように

#### ギャップ

- したがって、この要望は以下のどれかとして再定義する必要がある。

  2. 複数画像を表示したい
  3. タイムライン上での見え方を改善したい

#### 実装方針（調査結果を踏まえた候補）

### 4. 横位置が時系列になっていない問題を修正

#### ギャップ
- 設計上は `created_at` に基づく right anchor 配置で時系列になる想定。
- それでも問題が見えるなら、表示上の錯覚ではなく、配置式または周辺ロジックにズレがある可能性がある。

#### 原因仮説（優先度順）

##### 仮説1: ノートが「右端アンカー」で、可変幅のため左端基準で見ると時系列が崩れて見える
- 現状の note は `right: {p.f * 100}%` で配置され、カードは左に伸びる。
- 幅が短い投稿/長い投稿/画像付き投稿で左端位置が大きく変わるため、ユーザーが**カード左端**や**カード中心**で順序を認識すると「時系列になっていない」ように見えやすい。
- 特に画像つきカードは本文だけのカードより実寸が大きく、見た目のズレが増える。

##### 仮説2: レーン予約ロジックの幅見積もりが本文中心で、画像による実寸増加を織り込めていない
- `Timeline.svelte` の `estNotePx()` は author + content テキストを主に使って幅見積もりしている。
- 画像の存在はこの見積もりに反映されていない。
- そのため、同一レーンでの busy 区間が短く見積もられ、見た目上カード同士の前後関係が崩れたり、密集して見える可能性がある。

##### 仮説3: `created_at` が秒精度のため、同一秒の複数投稿で安定した横並び規則が不足している
- Nostrの `created_at` は秒精度。
- 同一秒に複数ノートがあると、横位置は完全一致する。
- その状態で lane 配置だけ異なると、時系列順が崩れたように見える。
- 安定ソート規則（id、受信順、元配列順）や同秒オフセット戦略を持たないと、視覚上の曖昧さが残る。

##### 仮説4: `windowMs` に対する比率配置自体は正しいが、ノート/軸の視覚基準が一致していない
- TimeAxis は tick を `right` 基準で打っている一方、ノートはカード本体の実寸を持つ。
- playhead line / tick / note anchor の認識点がUI上で明示されていないと、「軸に対してズレている」と感じやすい。

##### 仮説5: 受信順と挿入順は created_at 昇順だが、遅延到着イベントで体感と時系列が食い違う
- 古いイベントが遅れて到着すると、画面内に後から差し込まれる。
- データとしては正しいが、「今来たのに左に出た」ことをバグと認識される可能性がある。
- 必要なら「到着順」と「event time」の違いをUI上で補助する余地がある。

#### 実装方針
- まず原因切り分けを行う。
  1. 右端アンカーの視覚違和感か
  2. 幅見積もり不足か
  3. 同秒イベント問題か
  4. 画像付きカードでのみ悪化するか
- 修正方針候補:
  - ノートの**時刻アンカー点を視覚的に明示**する（小さなピン/縦線/尾）
  - 左端ではなく「時刻点」が分かるカードデザインへ変更
  - 同一秒イベントに微小オフセット規則を設ける
  - `estNotePx()` に画像有無/サムネイル幅を反映する
  - 必要なら衝突回避ロジックを「見た目実寸ベース」に寄せる

#### 影響ファイル候補
- `src/lib/components/Timeline.svelte`
- 必要なら `src/lib/nostr/images.ts`
- 必要なら `src/lib/timeline/store.svelte.ts`（同秒イベントの扱い補助をstateに持つ場合）
- 必要なら `src/lib/components/TimeAxis.svelte`（視覚的基準合わせ）

#### 注意点
- これは見た目だけの微調整ではなく、**UIの時間表現モデル**に触る可能性がある。
- 画像対応と密接に関係する。画像付きカードを増やすなら、横位置問題を先に/同時に直した方が手戻りが少ない。

## 動作確認方針

### 1. njump遷移
- ノートクリックで新規タブが開くこと
- 開くURLが期待する note/nevent を指すこと
- 既存の全文表示・TTSミュート導線が失われていないこと
- キーボード操作時の挙動（Enter/Space）確認

### 2. 50倍速
- 1× / 20× / 50× で playhead の進み方が破綻しないこと
- 再生中に LIVE / seek / nudge / pause を切り替えても壊れないこと
- TTS on/off のときに異常なキュー詰まりやクラッシュがないこと

### 3. 画像表示
- `imeta` タグつき投稿で表示されること
- `url` + `mime` タグ系で表示されること
- 本文内直リンク画像で表示されること
- 壊れたURLでカード全体が崩れないこと
- 複数画像対応を入れる場合は枚数・順序・トリミングを確認

### 4. 横位置時系列
- 同一ウィンドウ内で古い投稿ほど左、新しい投稿ほど右であること
- 長文/短文/画像付きで視覚的に逆転して見えないこと
- 同一秒投稿が複数あるケースで表示規則が安定していること
- 異なるウィンドウ幅（1min/5min/1hour）でも違和感が増えないこと

### 実施コマンド候補
- `npm run check`
- `npm run build`
- ブラウザ実機での目視確認（とくに Timeline の見え方）
