### 現状の各要望との対応状況

#### 1. 投稿クリックで njump を開く
- 現状、ノート本体クリックは `Timeline.svelte` の `openMenu(note)` に接続されている。
- クリックするとその場で外部遷移するのではなく、オーバーレイメニューを開く仕様。
- 要望の本質は **`https://njump.me/<note1...>` を別タブで開くこと**。単なる njump トップページ遷移ではなく、**対象投稿の `note1` を後ろに連結した個別投稿ページ遷移** が必要。
- したがって「njump導線が未実装」ではなく、**現状クリック挙動が要望仕様にまだ一致していない** と整理するのが正しい。

#### 2. 50倍速対応
- `src/App.svelte` の `SPEED_OPTIONS` は `[1, 1.5, 2, 3, 5, 8, 10, 15, 20]`。
- `timeline.setSpeed()` 側は `n > 0` の数値であれば受け入れるため、**状態管理層は20×固定ではない**。
- 現状の上限は実質的に **UIの選択肢が20×まで** で決まっている。
- 高速化に伴う副作用確認（playhead進行、TTS、視認性）は未整理。

#### 3. 投稿画像表示
- `Timeline.svelte` ではすでに `getNoteImageUrl()` を使って画像を描画している。
- モーダルにも画像プレビューがあり、画像クリックで別タブ表示できる。
- `src/lib/nostr/images.ts` では以下を検出している。
  - NIP-92 `imeta`
  - NIP-94系 `url` + `m` / `mime`
  - 本文中の直リンク画像URL
- つまり**「画像を表示できるように」は完全未着手ではなく、すでに一次実装済み**。
- ただし、現要望の意図次第では以下の不足がありうる。
  - 1枚のみ表示で、複数画像に未対応
  - 画像検出対象が限定的
  - レイアウト上、画像付きノートの横幅見積もりに画像寸法が反映されていない
  - 壊れた画像/重い画像/NSFW等の扱い未整理

#### 4. 横位置が時系列になっていない
- タイムライン上の横位置は `Timeline.svelte` で `f = (playhead - ms) / win` を計算し、`style="right: {p.f * 100}%"` で配置している。
- ノート配列 `timeline.notes` は `store.svelte.ts` で `created_at` 昇順に挿入され、`visibleNotes` もその順序を維持している前提。
- そのため、理屈上は **created_at が正しければ右（新）→左（旧）の時系列整列になる設計**。
- それでもズレが見える場合、原因は「並び順そのもの」よりも、**見た目上のアンカー位置・幅・同一時刻イベント・レーン予約ロジック** にある可能性が高い。

### 4. 横位置が時系列になっていない問題を修正

#### ギャップ
- 設計上は `created_at` に基づく right anchor 配置で時系列になる想定。
- それでも問題が見えるなら、表示上の錯覚ではなく、配置式または周辺ロジックにズレがある可能性がある。

#### 原因仮説（優先度順）

##### 仮説1: ノートが「右端アンカー」で、可変幅のため左端基準で見ると時系列が崩れて見える
- 現状の note は `right: {p.f * 100}%` で配置され、カードは左に伸びる。
- 幅が短い投稿/長い投稿/画像付き投稿で左端位置が大きく変わるため、ユーザーが**カード左端**や**カード中心**で順序を認識すると「時系列になっていない」ように見えやすい。
- 特に画像つきカードは本文だけのカードより実寸が大きく、見た目のズレが増える。

##### 仮説2: レーン予約ロジックの幅見積もりが本文中心で、画像による実寸増加を織り込めていない
- `Timeline.svelte` の `estNotePx()` は author + content テキストを主に使って幅見積もりしている。
- 画像の存在はこの見積もりに反映されていない。
- そのため、同一レーンでの busy 区間が短く見積もられ、見た目上カード同士の前後関係が崩れたり、密集して見える可能性がある。

##### 仮説3: `created_at` が秒精度のため、同一秒の複数投稿で安定した横並び規則が不足している
- Nostrの `created_at` は秒精度。
- 同一秒に複数ノートがあると、横位置は完全一致する。
- その状態で lane 配置だけ異なると、時系列順が崩れたように見える。
- 安定ソート規則（id、受信順、元配列順）や同秒オフセット戦略を持たないと、視覚上の曖昧さが残る。

##### 仮説4: `windowMs` に対する比率配置自体は正しいが、ノート/軸の視覚基準が一致していない
- TimeAxis は tick を `right` 基準で打っている一方、ノートはカード本体の実寸を持つ。
- playhead line / tick / note anchor の認識点がUI上で明示されていないと、「軸に対してズレている」と感じやすい。

##### 仮説5: 受信順と挿入順は created_at 昇順だが、遅延到着イベントで体感と時系列が食い違う
- 古いイベントが遅れて到着すると、画面内に後から差し込まれる。
- データとしては正しいが、「今来たのに左に出た」ことをバグと認識される可能性がある。
- 必要なら「到着順」と「event time」の違いをUI上で補助する余地がある。

#### 実装方針
- まず原因切り分けを行う。
  1. 右端アンカーの視覚違和感か
  2. 幅見積もり不足か
  3. 同秒イベント問題か
  4. 画像付きカードでのみ悪化するか
- 修正方針候補:
  - ノートの**時刻アンカー点を視覚的に明示**する（小さなピン/縦線/尾）
  - 左端ではなく「時刻点」が分かるカードデザインへ変更
  - 同一秒イベントに微小オフセット規則を設ける
  - `estNotePx()` に画像有無/サムネイル幅を反映する
  - 必要なら衝突回避ロジックを「見た目実寸ベース」に寄せる

#### 影響ファイル候補
- `src/lib/components/Timeline.svelte`
- 必要なら `src/lib/nostr/images.ts`
- 必要なら `src/lib/timeline/store.svelte.ts`（同秒イベントの扱い補助をstateに持つ場合）
- 必要なら `src/lib/components/TimeAxis.svelte`（視覚的基準合わせ）

#### 注意点
- これは見た目だけの微調整ではなく、**UIの時間表現モデル**に触る可能性がある。
- 画像対応と密接に関係する。画像付きカードを増やすなら、横位置問題を先に/同時に直した方が手戻りが少ない。

# 追補設計（2026-06-19）: タイムライン非重なりカードレイアウトエンジンの再設計

> この節は**設計のみ**。コードは変更しない。`Timeline.svelte` / `store.svelte.ts` の現状読込を踏まえ、
> カードが重ならず・下端からはみ出さず・同時刻でも決定的に並び・再生中も位置が暴れない
> レイアウト基盤への作り替えを定義する。小手先のヒューリスティック追加ではなく**レイアウトエンジンの再設計**を本命とする。

## 1. 現状のレイアウト方式と、なぜ今も破綻するか

### 1.1 現状方式（実装の事実）
- 配置の入力は store の `visibleNotes`（`playheadMs - windowMs <= ms <= playheadMs` でフィルタ、`(created_at, id)` 昇順）。
- 横位置のみ時刻で決まる: `f = (playhead - ms) / win`、カード**左端**を `left: (1 - f) * 100%` に置く（右端=playhead=新、左へ伸びる）。
- 縦は 6 レーン固定（`LANES = 6`）。レーンは identity-keyed でキャッシュ（`laneByNote`）し、一度割り当てたレーンは生涯不変＝ウィンドウが流れても縦に跳ねない。`laneByAuthor` で著者の前回レーンを優先。`feedVersion` 変化でキャッシュを破棄。
- レーン占有は **1次元の時間区間予約**で近似:
  - 幅は `estNotePx()`（author + content のテキスト幅をCJK判定 `isWide()` で推定し、CSS max にクランプ）＋ `cushion`。
  - それを `busyMs = win * min((estPx + cushion) / containerW, 1)` に変換し、`laneFreeAt[lane] = ms + busyMs` で「このレーンはこの時刻まで埋まっている」と予約。
  - 画像カードは `IMG_LANE_SPAN = 2` レーン縦に占有し、`chooseLane` で開始レーンを `0..LANES-span` に制限（＝下端はみ出し防止）し、span 全レーンが空くまでブロックを「空き」と見なさない。
  - 計測前（`containerW === 0`）は `FALLBACK_BUSY_FRACTION = 0.2` を使用。

### 1.2 なぜ今も破綻するか（根本原因）
現状は**「2次元（x幅 × y高さ）の矩形衝突」を「1次元の時間区間予約」で近似**しているため、近似が実寸とずれた瞬間に破綻する。具体的には:

1. **busyMs はDOM計測前・画像ロード前の幅推定に依存する**
   - `busyMs` は実レンダリング前の `estNotePx()` 推定から算出される。`containerW` 計測前は `0.2` 固定でさらに粗い。
   - 推定が実幅より小さいと予約区間が短くなり、隣カードが食い込んで**重なる**。大きいと過剰予約で**無駄な余白**になる。
   - しかも `estNotePx` は単一行の文字幅推定であり、`white-space: pre-wrap` の折返し・URL・絵文字・3行クランプの実挙動を再現しない。

2. **可変テキスト高さがモデル化されていない**
   - テキストカードは CSS `max-height: calc((100%/6) - 8px)` で1レーンに**クリップ**されるだけ。実際の内容高さ（1〜3行）は配置計算に一切入っていない。
   - 「1レーンに収まる」のは clamp による視覚的切詰めであって、収まっている保証ではない。`head-row`（`flex: 0 0 auto`）＋3行 content が1レーン高さを超えると、下端が削れて隣レーンに視覚的に接触/侵食しうる。

3. **可変カード幅が実測されていない**
   - `width: max-content; max-width: min(340px, 60vw)` なので実幅はブラウザ計測でしか確定しない。推定 `estNotePx` との差がそのまま衝突/余白の誤差になる。

4. **画像カードの実寸が不定**
   - 画像の真のアスペクト比はロードまで不明。`IMG_LANE_SPAN = 2` と CSS `.note.has-image max-height: calc((100%/6)*2 - 8px)`、`.note-image height: 60vh` の縮小トリックで「2レーンに押し込む」が、**予約フットプリントは実寸と無関係な固定2レーン推定**。横幅も `IMG_NOTE_PX = 240` という floor 推定にすぎない。
   - 画像が後から settle するとカード実寸が変わり、配置済みの近似と乖離する（ロード前後でのレイアウト揺れ）。

5. **同一秒ノート（秒精度 created_at）**
   - `f` が同値→ `left%` が完全一致。横方向では区別できず、違いはレーン（縦）だけ。空きレーン数を超えて同秒ノートが集中すると、`chooseLane` は「最も早く空くブロック」にフォールバックし、**まだ埋まっているレーンへ重ねて置く**。
   - 順序自体は store の `(created_at, id)` 安定ソートで決定的だが、横位置が一致するため衝突回避を縦に頼り切っており、レーンが尽きると重なる。

6. **下端オーバーフロー制約の脆さ**
   - 下端非はみ出しは「画像開始レーン <= `LANES - IMG_LANE_SPAN`」と CSS 高さの**手動一致**に依存している（コメントにも「片方だけ変えると重なりが再発する」と明記）。`IMG_LANE_SPAN` と CSS が二重管理で、不変条件がコードとスタイルに分散している。
   - テキストカードも、実高さが1レーンを超えた場合に最下レーンで下端を割る可能性が原理的に残る。

7. **貪欲レーン割当 + busyMs 近似の限界（総括）**
   - 貪欲＋不変キャッシュは「縦跳ね防止（安定性）」には有効だが、`busyMs` は**幅と時間を1次元に潰した予約**であり、実測矩形どうしの真の2D衝突を表現できない。
   - 近似が実DOM寸法から乖離する限り、どんなに `cushion` / `IMG_BUFFER_PX` を盛っても「重ならない保証」にはならない（バッファは確率的緩和にすぎない）。
   - 結論: **本質的に「推定値ベースの1D区間予約」を捨て、「実測矩形ベースの2D配置」に作り替える必要がある。**

## 2. 必須制約 / 不変条件

レイアウトエンジンは以下を**保証**しなければならない（緩和ではなく不変条件として）:

- **C1 非重なり**: 表示中の任意の2カードのレンダリング矩形（x区間 × y区間、`cushion` 込み）が交差しない。
- **C2 下端（上端）非はみ出し**: すべてのカードがタイムライン可視領域の縦範囲 `[0, H]` に完全に収まる（`y + height <= H`、`y >= 0`）。画像カードも例外なし。
- **C3 同時刻の決定的順序**: 同一 `created_at` のノートは `(created_at, id)` の安定順（store 既存）で処理し、配置結果が実行ごとに同一。
- **C4 再生中の位置安定**: 一度割り当てた縦位置はそのノートの生存中不変。ウィンドウが流れても・playhead が毎フレーム進んでも縦に跳ねない。横位置のみ時刻に従って連続移動する。
- **C5 時刻真実性（横は時刻専用）**: 横位置は常に `left = (1 - f)` であり、レイアウトエンジンは**縦方向（レーン／サブレーンオフセット）でのみ**衝突を解消する。横を時刻からずらして重なりを避けることは禁止（時系列表現が壊れるため）。
- **C6 純関数性**: 配置結果は `(visibleNotes, playheadMs, windowMs, 実測サイズ, containerW/H)` の純関数。副作用・乱数・到着順依存を持たない（C3/C4 を支える）。
- **C7 計測前後の単調収束**: 実測が未確定な初回フレームでも決定的な事前見積りで配置し、実測（または画像 settle）到着時に**同じ決定的規則**で再配置して収束する。再配置はノート単位の縦位置のみ更新し、横はいじらない。

## 3. アルゴリズム候補と比較

| 案 | 概要 | 非重なり保証 | 可変高さ | 画像実寸 | 同秒衝突 | 安定性 | 計算量 | 評価 |
|----|------|------------|---------|---------|---------|--------|--------|------|
| **A. 現状改良（1D区間予約＋実測幅）** | `busyMs` を残しつつ幅だけ実測値/画像 intrinsic に差替え | △（縦高さを表現できず） | × | △ | △ | ○ | O(n) | 重なりの根本（高さ・2D）が残る |
| **B. スカイライン/シェルフ 2Dパッキング（実測矩形）** | 各カードを実測矩形とし、x区間が重なる active 矩形群に対し最下の非衝突 y を探索 | ○ | ○ | ○ | ○（縦で確実に分離） | ○（id キャッシュ） | O(n·k)（k=同時 active 数, 実質小） | **本命** |
| **C. 区間グラフ彩色（sweep-line, 実測区間）** | x をスイープし active 集合を保ち、可変高さを「必要レーン数」に量子化して割当 | ○ | ○（量子化） | ○ | ○ | ○ | O(n log n) | B の離散版。実装容易だがレーン量子化で粒度損 |
| **D. 力学/バネ（force layout）** | 重なりカードを縦に反発で押し分ける | △（収束保証なし） | ○ | ○ | △ | ×（毎フレーム揺れ） | 反復 | **却下**（非決定・ジッタ、C3/C4違反） |
| **E. 2フェーズ事前計測＋決定的パッカー** | オフスクリーン/初回レンダで実測 → キャッシュ → B/C の決定的パッキング | ◎ | ◎ | ◎ | ◎ | ◎ | O(n log n) | B を計測戦略で補強した完成形 |

補足:
- A は最小変更だが C1/可変高さ（C 高さ方向）を満たせず、要件の核心（重なり/下端はみ出し）を解けない。
- D は非決定でC3・C4を破るため不採用。
- B と E は排他ではない。**「決定的パッカー（B/C）＋計測駆動（E）」を統合**したものが推奨案（第4節）。

## 4. 推奨アプローチ（決定的・計測駆動の2Dスイープライン・パッカー）

**結論: 1D `busyMs` 予約を廃し、実測矩形に対する決定的スイープライン2Dパッカー（案B＋C＋E統合）へ作り替える。** 固定6レーンは可変高さに対応できないため、縦を「連続pxオフセット（または十分細かい量子レーン）」に一般化する。

### 4.1 入力となるカード矩形の確定（高さ・幅は実測/キャッシュ由来）
- 各カードの `width` / `height` は**テキストヒューリスティックで推定せず、計測またはキャッシュから得る**ことを原則とする:
  - 一次ソース: 描画後の `getBoundingClientRect()`（または `ResizeObserver`）で実測し、`note.id + content署名 + 画像ロード状態 + containerW` をキーに**計測キャッシュ**へ保存。
  - 画像カードは画像 intrinsic（`naturalWidth/Height`）が確定した時点で矩形を更新し、再パッキングをトリガ（C7）。
  - 代替（推奨補助）: **決定的事前計測（オフスクリーン測定）** — 同一CSSの隠し要素に流し込んで `(w,h)` を確定してから配置する。実測のちらつきを避けたい場合の決定的経路。
  - フォールバック: 計測前の初回のみ、現行 `estNotePx()` 系の事前見積りで矩形を仮置き（C7 の収束前段）。ただしこれは**初期値であって最終配置の根拠にしない**。
- 矩形の x 区間は時刻から: `x0 = (1 - f) * W_px`、`x1 = x0 + width + GAP`（`cushion` を `GAP` に統合）。横は時刻専用（C5）。

### 4.2 決定的スイープラインによる縦配置
- `visibleNotes` を `(created_at, id)` 昇順（store 既存順）で走査（C3）。
- active 集合 = 「x 区間が現在カードと重なる既配置カード」。新カードに対し:
  1. **既に id キャッシュに縦位置を持つなら再利用**（C4: 生涯不変）。新規のみ割当。
  2. 縦位置候補を下（y=0）から探索し、active 集合のどの矩形とも `[x0,x1] × [y, y+height]` が交差しない最小の `y` を選ぶ（C1）。
  3. `y + height <= H`（可視高さ）を満たす範囲に限定（C2: 下端はみ出し禁止）。満たす `y` が無い場合は後述「オーバーフロー方針」。
  4. 著者前回位置（`laneByAuthor` 相当）を「同点時の優先ヒント」として残してよいが、不変条件を破らない範囲に限定。
- 縦は連続pxでも、実装簡素化のため「`min-card-height` 単位の細粒度量子レーン（例: 高さ/12 など現行6より細かい格子）」でもよい（案C）。量子化する場合も**実測高さから必要レーン数を算出**する点が現行との本質差。

### 4.3 オーバーフロー方針（C2 を不変に保つ）
- 縦に置き場が無い（混雑）場合でも C1/C2 を破ってはならない。許容する縮退は以下のいずれかを設計時に選択:
  - (a) 同 x で置けない最古カードを当該フレームで**非表示にデフ**（描画から落とす。順序は決定的）。
  - (b) カード高さに**ハードな上限**を設け（テキストはクランプ強化、画像は最大フットプリントを制限）、最大同時表示数を H から逆算して保証。
  - (c) 混雑時のみ画像カードのフットプリントを段階縮小（ただし決定的閾値で）。
- いずれも「重ねて回避」は禁止（C5 で横ずらしも禁止）。**安全側 = 落とすか縮める**。

### 4.4 CSS とレイアウト計算の脱結合
- 現状の `IMG_LANE_SPAN` ↔ CSS `max-height` 二重管理を解消する。**カードの縦サイズは JS の配置結果（計測値）を CSS 変数/インラインスタイルへ一方向に流す**（例: `style="top: {y}px; height: {h}px"`）。
- これにより「片方だけ変えると重なる」脆さ（現状コメントの警告）を構造的に除去する。CSS は見た目専用、寸法の真実はレイアウトエンジン側に一本化。

## 6. 実装順序

1. **純パッカー `layout.ts` を先に作る（DOM非依存）**
   - 入力: `{id, x0, width, height}[]` ＋ `H` ＋ 既存縦位置キャッシュ。出力: `{id, y}[]`。
   - ここで C1/C2/C3/C4/C6 をユニットテストで固める（DOMなしで検証可能）。
2. **計測層の導入**
   - 実測（`getBoundingClientRect`/`ResizeObserver`/画像 `onload`）→ 計測キャッシュ。初回は事前見積りで仮置き、計測到着で再パッキング（C7）。
3. **Timeline.svelte の配置を差替え**
   - `busyMs`/`chooseLane`/固定6レーン/`IMG_LANE_SPAN`-CSS結合を撤去し、`layout.ts` 結果を `top/height` インラインへ。
   - `laneByNote` の役割は `layout.ts` の「id→y キャッシュ」へ移管。`feedVersion` リセットは踏襲。
4. **オーバーフロー縮退方針の実装（4.3 から選択）**
5. **画像 settle 時の再パッキング配線と、揺れ抑制（事前計測 or アスペクト確定までプレースホルダ寸法）**
6. **CSS 整理**: 寸法系を JS 駆動に寄せ、装飾のみ CSS に残す。

## 8. pause/LIVE セマンティクスと現状 store モデルの整合

- **純関数として成立する（C6）**: 配置は `visibleNotes`（store 既存の純 `$derived` フィルタ）＋計測寸法＋H の関数。`playheadMs` は**横位置 `f` の連続移動にのみ**効き、縦割当には効かない（縦は id キャッシュで固定）。よって毎フレームの playhead 進行で再パッキングは不要——縦は安定、横だけ滑らかに動く。
- **LIVE**: `#step()` が毎フレーム `playheadMs = now` を更新するが、可視集合が変わらない限り縦配置は不変。`aiContextSig`（count + first/last id）に倣い、**「可視集合の署名」変化時のみ縦パッキングを再計算**し、横は毎フレーム `f` 再計算でよい（軽量）。新ノート流入時のみ新規 id に縦を割当。
- **pause（LIVEからの一時停止）**: `pause()` は `isLive=false` かつ `playheadMs` を現在端に固定し `#step` を no-op 化する。playhead が止まるので**横も縦も完全静止** → レイアウトは静的スナップショットになり、検証も容易。
- **seek / nudge / jumpTo / goLive**: いずれも可視集合が入れ替わるが、`feedVersion` 不変なら同一ノートの縦キャッシュは維持され、再表示時も同じ `y`（C4）。フィード再構築（`#clearNotes` → `feedVersion++`）時のみ縦キャッシュを破棄＝新レイアウト、という現行契約をそのまま使う。
- **決定的順序は store が既に保証**: `addNote` の `(created_at, id)` 二分挿入が C3 の入力順を与える。遅延到着で古いノートが差し込まれても、順序は決定的で、パッカーは既存縦位置を保ったまま新ノートにのみ空き `y` を割り当てる（既存カードを動かさない）。
- **MAX_NOTES キャップ**: playhead から遠い端を落とす現行挙動はレイアウトに影響しない（可視窓外）。可視窓内の縦キャッシュ寿命は `feedVersion` と id 存在に紐づけて管理する。
